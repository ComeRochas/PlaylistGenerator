\section{Introduction}
% ─────────────────────────────────────────────────────────────────────────────

Music streaming platforms host hundreds of millions of tracks, yet their recommendation
engines still rely largely on collaborative filtering and rigid keyword tags. When a listener
wants ``\textit{chill jazz for relaxing}'' or ``\textit{upbeat reggae with rhythmic percussion}'',
a keyword search often fails to match the intended mood, because natural language encodes
semantic nuance that traditional metadata cannot capture.

This project, \textbf{PlaylistGenerator}, addresses this gap by building a semantic music retrieval
system that maps both audio and free-text queries into a shared embedding space. Given
any natural-language prompt, the system returns a ranked playlist of the most
\emph{semantically} similar tracks in the corpus. An additional research direction extends
this to \emph{visual} prompts: a user-provided image can be projected into the same embedding
space, enabling image-to-playlist queries.

The system is built on three components:
\begin{enumerate}
    \item \textbf{Audio corpus indexing} --- pre-compute CLAP audio embeddings for every
          track in the FMA Small dataset and store them in a compressed index.
    \item \textbf{Text-to-playlist retrieval} --- encode the user's text prompt with the
          CLAP text encoder and run a cosine $k$-nearest-neighbor search against the index.
    \item \textbf{Domain adaptation} --- fine-tune the CLAP text encoder with LoRA on the
          MusicCaps dataset to better align music-specific captions with audio.
\end{enumerate}
A fourth, exploratory component trains a small MLP bridge from CLIP image features into
the CLAP text space, opening the path toward image-conditioned playlist generation.

\section{Background}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{Contrastive Language–Audio Pre-training (CLAP)}

CLAP~\cite{elizalde2023clap} is a dual-encoder model that maps audio clips and text
descriptions into a shared 512-dimensional embedding space via contrastive learning.
The audio branch uses an \textbf{HTSAT-base} hierarchical Swin transformer, while the
text branch is \textbf{RoBERTa}. During pre-training, matched audio--text pairs are
pulled together while mismatched pairs are pushed apart using the
\textbf{symmetric InfoNCE loss}:
\begin{equation}
    \mathcal{L} = \frac{1}{2}\Bigl(
        \mathcal{L}_{\text{audio}\to\text{text}}
      + \mathcal{L}_{\text{text}\to\text{audio}}
    \Bigr), \quad
    \mathcal{L}_{a\to t} = -\frac{1}{N}\sum_{i=1}^{N}
        \log\frac{e^{s\,\mathbf{a}_i^\top\mathbf{t}_i}}
                  {\sum_{j=1}^{N} e^{s\,\mathbf{a}_i^\top\mathbf{t}_j}}
    \label{eq:infonce}
\end{equation}
where $s = \exp(\log\_\text{scale})$ is a learned temperature and $N$ is the batch size.
The checkpoint used throughout this project is
\texttt{music\_audioset\_epoch\_15\_esc\_90.14.pt}, pre-trained on music and AudioSet data,
achieving 90.14\% accuracy on ESC-50.

\subsection{CLIP for Vision}

CLIP~\cite{radford2021clip} follows the same contrastive paradigm for image--text pairs.
We use the \textbf{ViT-L/14} variant (\texttt{openai/clip-vit-large-patch14}), which
produces 1024-dimensional image embeddings. By training a bridge network from CLIP's
image space to CLAP's text space, we can query the audio index with images instead of text.

\subsection{Parameter-Efficient Fine-tuning with LoRA}

LoRA~\cite{hu2022lora} inserts low-rank adapters into the frozen weight matrices of a
pre-trained model. For a weight matrix $W \in \mathbb{R}^{d \times k}$, the adapted
forward pass becomes:
\begin{equation}
    h = W x + \Delta W\,x = W x + B A\, x,
    \quad B \in \mathbb{R}^{d \times r},\; A \in \mathbb{R}^{r \times k},\; r \ll \min(d,k).
    \label{eq:lora}
\end{equation}
Only $A$ and $B$ are trained; the original weights $W$ remain frozen.
With $r=16$ and $\alpha=32$, injecting LoRA into the \texttt{query} and \texttt{value}
projections of RoBERTa yields only \textbf{1,246,209 trainable parameters}, out of
200,154,457 total (0.62\%).

\section{Datasets}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{FMA Small}

The \textbf{Free Music Archive (FMA) Small} subset~\cite{defferrard2017fma} contains
\textbf{8,000 tracks} across 8 balanced genres (Hip-Hop, Pop, Folk, Experimental,
Rock, International, Electronic, Instrumental), each 30-second clip at 44.1 kHz.
The archive weighs approximately 7.2 GiB. Out of 8,000 files, 6 are corrupt MP3s
(silently skipped during embedding), yielding \textbf{7,994 indexed tracks}.
This corpus constitutes the retrieval database.

\subsection{MusicCaps}

\textbf{MusicCaps}~\cite{agostinelli2023musiclm} is a music captioning benchmark
released by Google and hosted on Hugging Face (\texttt{google/MusicCaps}).
It provides 5,521 10-second YouTube clips paired with expert free-text captions
describing instrumentation, tempo, mood, and genre. After downloading with
\texttt{yt-dlp}, 5,298 clips were successfully retrieved (223 permanently
unavailable). Pre-computing audio embeddings from the frozen CLAP audio encoder
and pairing them with captions produced \textbf{4,357 usable samples}, split
85\,/\,15 into \textbf{3,704 training} and \textbf{653 test} samples (seed 42).
This split is used for LoRA fine-tuning.

\subsection{MS COCO Captions}

The \textbf{MS COCO Captions} dataset provides image--caption pairs for the
image-conditioned bridge experiment. We sample the first \textbf{20,000 training
examples} (shuffled, seed 42) from the Hugging Face \texttt{HuggingFaceM4/COCO}
dataset. Each sample provides one image (encoded by the frozen CLIP ViT-L/14) and
one caption (encoded by the frozen CLAP text encoder), yielding paired
\((v_{\text{CLIP}}, e_{\text{CLAP}})\) tuples for bridge MLP training.

\subsection{Audio Corpus Indexing}

All 8,000 MP3 files in the FMA Small dataset are processed with the CLAP audio encoder
(\texttt{compute\_embeddings.ipynb}). Files are loaded at 22050 Hz, passed through the
\texttt{CLAP\_Module.get\_audio\_embedding\_from\_filelist} method, then L2-normalised.
Six corrupted files raise a \texttt{NoBackendError} and are silently skipped. The resulting
\textbf{7,994 vectors of dimension 512} are stored in a compressed NumPy archive
(\texttt{data/embeddings.npz}), along with their file-path keys, constituting the
\emph{audio index}.

\subsection{Text-to-Playlist Retrieval}

At inference time (\texttt{utils.get\_playlist}), the user's natural language query is
encoded by the CLAP text encoder and L2-normalised to produce a query vector
$e_q \in \mathbb{R}^{512}$. Cosine similarity with all corpus embeddings reduces to a
matrix--vector product:
\begin{equation}
    \text{sim}(i) = e_q^\top e_i, \quad e_i \text{ already L2-normalised.}
    \label{eq:cosine}
\end{equation}
The top-$k$ indices are extracted in $\mathcal{O}(N)$ via \texttt{np.argpartition},
then sorted. The function returns a list of \texttt{(score, filepath)} pairs.

\subsection{LoRA Fine-tuning on MusicCaps}

The pre-trained CLAP model was trained on general audio--text pairs, so its text encoder
may underperform on music-specific captions. We fine-tune \emph{only the text encoder}
(RoBERTa) using LoRA while keeping the audio encoder \textbf{fully frozen}.

\paragraph{Adapter configuration.}
\texttt{LoraConfig(r=16, lora\_alpha=32, lora\_dropout=0.1,
target\_modules=["query","value"])} is applied to \texttt{model.model.text\_branch}
via PEFT, yielding 1,246,209 trainable parameters (0.62\% of 200M).

\paragraph{Training procedure.}
Audio embeddings are pre-computed once from the frozen audio encoder and cached as
\texttt{data/musiccaps\_embeddings.npz}. Each training step loads a batch of
\textit{(cached audio embedding, caption)} pairs; the text encoder encodes the captions
to produce $e_t$, and the symmetric InfoNCE loss (Eq.~\ref{eq:infonce}) is computed
between the batch of audio embeddings $e_a$ and text embeddings $e_t$.
Hyperparameters are summarised in Table~\ref{tab:hparam}.

\subsection{\texorpdfstring{CLIP$\to$CLAP}{CLIP-to-CLAP} Bridge MLP}

To support image-conditioned queries, a small MLP (\texttt{BridgeMLP}) is trained to
project 1024-dimensional CLIP image features into the 512-dimensional CLAP text space.
Both the CLIP and CLAP encoders are \textbf{fully frozen}; only the bridge is trained.
The architecture is:
\begin{align*}
    h_1 &= \text{Dropout}_{0.2}\!\left(\text{LayerNorm}\!\left(\text{ReLU}\!\left(W_1 x\right)\right)\right),
           \quad W_1 \in \mathbb{R}^{1024\times 1024} \\
    h_2 &= \text{Dropout}_{0.2}\!\left(\text{LayerNorm}\!\left(\text{ReLU}\!\left(W_2 h_1\right)\right)\right),
           \quad W_2 \in \mathbb{R}^{1024\times 1024} \\
    \hat{e} &= W_3 h_2, \quad W_3 \in \mathbb{R}^{1024\times 512}.
\end{align*}

Training uses a one-directional InfoNCE loss at temperature $\tau = 0.07$ on 20,000
MS COCO caption pairs, with AdamW ($lr=2\times10^{-4}$, $\lambda=10^{-2}$), batch
size 64, 5 epochs. The trained bridge is saved to
\texttt{artifacts/bridge\_mlp\_clip\_to\_clap.pt}.

\section{Conclusion and Future Work}
% ─────────────────────────────────────────────────────────────────────────────

This project demonstrates that a pre-trained CLAP model can serve as a practical
backbone for semantic music retrieval, and that lightweight LoRA fine-tuning on the
music-specific MusicCaps benchmark meaningfully improves alignment quality
(+11.5\% cosine similarity) while updating less than 1\% of the model's parameters.
The result is a working end-to-end system: given a free-text description, the user
receives a ranked playlist drawn from 7,994 FMA tracks within milliseconds.

Two main directions are planned for future work.

\paragraph{Image-to-playlist generation.}
The trained CLIP$\to$CLAP bridge MLP enables a new interaction modality: a user
uploads a photograph (a forest, a party scene, a painting) and the system projects
the CLIP image embedding into the CLAP text space, then queries the audio index
directly. The bridge is trained but not yet integrated into the end-to-end retrieval
pipeline; this integration is the immediate next step.

\paragraph{RLHF alignment.}
Cosine similarity in the CLAP space is a proxy for human preference, not a direct
measure of it. Reinforcement Learning from Human Feedback (RLHF) could be used to
further refine the retrieval ranking: users listen to suggested tracks, provide
positive/negative signals, and a reward model is trained to score playlists according
to human satisfaction. This would close the loop between the embedding-based retrieval
and actual listener preferences, making the system more personalised over time.
